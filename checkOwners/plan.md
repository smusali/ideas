## Execution Steps

Use the Claude Code `plan` command to plan each step before executing it.

- **Step 1: Project scaffold**
  - `hatch new checkowners` to generate the initial project structure
  - Write `pyproject.toml` with Typer, Rich, GitPython, PyGithub, PyYAML dependencies and hatch build config
  - Create the Typer CLI entry point in `checkowners/cli.py` with placeholder subcommands
  - Implement `config.py`: PyYAML loader for `.github/checkowners.yml` with defaults
  - Set up pytest harness with `hatch run test` script and `tests/` directory structure

- **Step 2: Git analysis engine**
  - Implement `models.py`: `OwnershipMap`, `DriftResult`, and `Config` dataclasses
  - Implement `analyze.py`: parse `git log` and `git blame` via GitPython and subprocess
  - Build the commit-to-path ownership map respecting `lookback_days`, `min_commits`, `top_n_owners`
  - Apply default path exclusions (`*.lock`, `dist/**`, `vendor/**`) from config
  - Write `tests/test_analyze.py` with mocked subprocess calls (no real git repo needed)

- **Step 3: CODEOWNERS generator**
  - Implement `generate.py`: normalize paths, select top-N owners, write `.github/CODEOWNERS`
  - Prepend the machine-generated header to every write: `# Generated by checkOwners. Do not edit manually.`
  - Support `include_unowned: true/false` config option for human triage paths
  - Write `tests/test_generate.py`

- **Step 4: Drift detection state machine**
  - Implement `drift.py`: compare `OwnershipMap` from `analyze.py` against current `.github/CODEOWNERS`
  - Detect stale entries, missing entries, and changed paths; output as `DriftResult`
  - Support all three state machine modes (`commit`, `repo`, `both`) from config
  - Output JSON diff via `--json` flag; write `GITHUB_OUTPUT` when running in Actions
  - Write `tests/test_drift.py`

- **Step 5a: GitHub Actions integration (before PyPI publish)**
  - Implement the `GITHUB_OUTPUT` writer in `drift.py` (activated by env var detection)
  - Write `action.yml` composite action with `uses: actions/checkout` + `pip install checkowners` + `checkowners drift` steps
  - Add an example `.github/workflows/checkowners-example.yml` to the repo showing real usage
  - Open the secondary Claude Code session (`cd checkowners-action && claude`) to place `action.yml` there
  - Validate `action.yml` syntax with `actionlint action.yml` (zero dependency on PyPI)
  - Run `hatch build` in the `checkowners` repo to produce a local wheel; temporarily point `pip install` at `dist/checkowners-*.whl` in `action.yml` and test end-to-end locally with `act`
  - Revert the local wheel reference back to `pip install checkowners` before committing

- **Step 5b: GitHub Actions end-to-end test (after PyPI publish + `v1` tag are live)**
  - Restore `pip install checkowners` in `action.yml` (resolves from the real registry)
  - Create the `v1` release tag on `checkowners-action`; submit to GitHub Marketplace
  - Reference `checkowners/checkowners-action@v1` from a real workflow and verify it resolves correctly
  - Do NOT commit `.github/workflows/checkowners.yml` to the `checkowners` repo until this passes (see "Publish Order" below)

- **Step 6: Notifications + validate + print**
  - Implement `notify.py`: HTTP POST to `webhook_url` from config on drift events; skip if `webhook_url` is empty
  - Implement `validate.py`: syntax-only CODEOWNERS parser; no inference; exits non-zero on invalid syntax
  - Implement `checkowners print` command: print inferred owners map to stdout (supports `--json`)
  - Write `tests/test_validate.py`

- **Step 7: Packaging, README, and smoke-test**
  - Write the full `README.md` with badges, install instructions, CLI reference, config reference, and dogfooding example
  - Run `hatch run test` and confirm 85%+ coverage across all modules
  - Run `hatch build` and `pip install dist/*.whl` in a fresh virtualenv to smoke-test
  - Run `checkowners analyze` and `checkowners drift` against the `checkowners` repo itself (dogfooding)

## Publishing and Distribution

### Publish Order (Critical)

The dogfooding CI workflow references `checkowners/checkowners-action@v1`, which internally runs `pip install checkowners`. This creates a hard dependency between the two publish steps and the CI wiring. The correct sequence:

1. Complete Steps 1-6 and locally smoke-test with the local wheel (Step 5a)
2. Complete Step 7: full README, 85%+ coverage, `hatch build`, local smoke-test
3. `hatch publish` to push `checkowners` to PyPI; tag `v0.1.0` on the `checkowners` repo
4. Run Step 5b: restore `pip install checkowners` in `action.yml`; create the `v1` release tag on `checkowners-action`; submit to Marketplace
5. Only after both steps 3 and 4 are complete: commit `.github/workflows/checkowners.yml` to the `checkowners` repo
6. Verify CI passes on the first push

Incorrect order: committing the dogfooding workflow before PyPI publish or before the `v1` action tag exists causes `pip install checkowners` to 404 on every CI run, breaking all CI until the race is resolved.
